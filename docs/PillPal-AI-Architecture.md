# PillPal AI 助手技术架构

> 基于 DeepSeek V3 的 ReAct Agent，通过 function calling 连接 14 个工具，实现"对话即操作"的用药管理体验。

---

## 整体流程

```
用户说话："阿莫西林吃了"
         │
         ▼
┌─────────────────────────────────┐
│  构建上下文（Context）             │
│  ├── 系统提示词（小派人设 + 规则） │
│  ├── 用户画像（年龄/疾病/药品）    │
│  ├── 实时数据（今日打卡/库存）     │
│  ├── 对话历史（或压缩摘要）        │
│  └── 14 个工具定义                │
└─────────────────────────────────┘
         │
         ▼  发送给 DeepSeek V3（流式 SSE）
┌─────────────────────────────────┐
│  ReAct 循环（最多 8 步）           │
│                                  │
│  Step 1: AI 判断需要打卡           │
│    → 返回 tool_call:              │
│      take_med("阿莫西林", [])      │
│                                  │
│  Step 2: 前端执行工具              │
│    → 写入数据库 → 扣减库存         │
│    → 返回 "已打卡：阿莫西林"       │
│                                  │
│  Step 3: AI 看到结果，生成回复      │
│    → 流式输出："打卡成功！💪…"     │
└─────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────┐
│  后处理                           │
│  ├── 刷新药品 / 记录数据           │
│  ├── 异步提取记忆（置信度 > 0.85） │
│  └── 语音播报（TTS 流式朗读）      │
└─────────────────────────────────┘
```

## 四层架构设计

### 第 1 层：用户画像（buildUserProfile）

把用户的基本信息、药品列表、AI 提取的长期记忆组装成一段文字，注入到系统提示词。这样 AI 每次回复都了解"你是谁、吃什么药、有什么特殊情况"。

**示例输出：**

```
【用户画像】小明，65岁，男
疾病：高血压、糖尿病
药品：
- 氨氯地平片（5mg × 1片），空腹，治高血压，08:00
- 二甲双胍片（0.5g × 1片），餐后30分钟，治糖尿病，08:00/18:00
【记忆】用户对青霉素过敏
```

**设计要点：**
- 每次对话前动态构建，保证信息是最新的
- 记忆来自 AI 自动提取（下文会讲），不需要用户手动维护

---

### 第 2 层：实时数据（buildRealtimeData）

每次对话前从数据库拉取当天的打卡状态和库存情况，让 AI 掌握当前上下文。

**示例输出：**

```
【当前时间】2026年5月15日 周四 14:30
【今日状态】3次药中已服1次
已服：氨氯地平(08:15)
待服：二甲双胍(08:00)、二甲双胍(18:00)
【库存】氨氯地平 28片/28天✅  二甲双胍 5片/2天⚠
```

**为什么这样设计：**
- 相比让 AI 每次先调工具查状态，直接注入数据能减少一轮 Agent 循环，响应更快
- 库存预警（≤7 天标 ⚠）让 AI 主动提醒用户补货

---

### 第 3 层：对话历史管理（manageHistory）

对话越来越长，token 会爆。采用**阈值压缩**策略：

```
对话 token < 4000  →  全部保留，原样发给 AI
对话 token > 4000  →  旧对话压缩成一段摘要 + 保留最近 6 轮完整对话
```

**压缩过程：**
1. 把对话按"一问一答"分成若干轮
2. 保留最近 6 轮不动
3. 更早的对话 + 已有摘要 → 调 DeepSeek 生成新摘要（约 150 tokens）
4. 摘要以 `[对话摘要]` 前缀放在历史开头

**好处：**
- 用户可以长时间聊天不丢失上下文
- 最近的对话保持完整细节，旧对话保留要点
- token 成本可控

---

### 第 4 层：ReAct Agent 循环（核心）

不是简单的一问一答，而是一个**推理-行动-观察**的循环：

```
while (步数 < 8) {
    把完整上下文发给 DeepSeek V3

    if (AI 返回 tool_calls) {
        → 并行执行所有工具
        → 工具结果作为 observation 放回对话历史
        → continue（让 AI 看到结果后继续推理）
    }

    if (AI 返回文本) {
        → 流式输出给用户
        → break（本轮结束）
    }
}
```

**关键设计：**

- **并行工具调用**：AI 可以一次请求多个工具（如同时查状态 + 查库存），前端 `Promise.all` 并行执行，减少等待
- **多步推理**：AI 看到工具结果后可能需要再调另一个工具（最多 8 步），支持复杂任务链
- **流式输出**：最终文本回复通过 SSE 逐字推送，前端实时显示打字效果

**示例：复杂问题的多步推理**

```
用户："帮我看看今天吃药情况，顺便查查库存"

Step 1: AI 判断需要两个数据
  → 并行调用 get_today_status + get_stock

Step 2: 两个工具同时执行，返回结果：
  - "今日3次药中已服1次，氨氯地平(08:15)已服..."
  - "氨氯地平 28片/28天✅  二甲双胍 5片/2天⚠"

Step 3: AI 综合两个结果，生成自然语言回复：
  → "今天 3 次药吃了 1 次，氨氯地平早上已经打卡啦 ✅
     二甲双胍上午和晚上的还没吃，记得按时服用哦~
     另外二甲双胍库存只剩 2 天了 ⚠️ 该补货了！"
```

## 14 个工具（Function Calling）

### 工具分类

| 类型 | 工具名 | 功能 |
|------|--------|------|
| **感知**（只读） | `get_today_status` | 今日各药/各时段的打卡状态 |
| | `get_stock` | 所有药品库存及可用天数 |
| | `get_medications` | 完整药品列表及详细信息 |
| | `get_history` | 过去 N 天的服药历史和依从率 |
| **执行**（写入） | `take_med` | 打卡记录服药 |
| | `skip_med` | 跳过某次服药（记录原因） |
| | `undo_med` | 撤回误操作的打卡 |
| | `update_stock` | 修改药品库存数量 |
| | `add_medication` | 添加新药到用药计划 |
| | `remove_medication` | 从计划中移除药品 |
| **知识 + 记忆** | `search_knowledge` | 搜索药学知识库 |
| | `update_memory` | 存储用户长期记忆 |
| | `generate_report` | 生成今日服药报告 |

### 工具定义方式

每个工具以 JSON Schema 格式定义，包含名称、描述、参数类型，随 API 请求一起发送给 DeepSeek：

```json
{
  "type": "function",
  "function": {
    "name": "take_med",
    "description": "帮用户打卡记录服药。当用户说吃了某个药...",
    "parameters": {
      "type": "object",
      "properties": {
        "med_name": { "type": "string", "description": "药品名称" },
        "time_slots": { "type": "array", "items": { "type": "string" } }
      },
      "required": ["med_name", "time_slots"]
    }
  }
}
```

AI 根据用户的自然语言，自动决定调用哪个工具、传什么参数。前端代码执行工具逻辑并将结果返回给 AI。

### 澄清机制（安全设计）

通过系统提示词约束 AI 的行为，防止误操作：

| 场景 | AI 行为 |
|------|---------|
| 用户说"打卡"但没说哪种药（有多种药时） | 先问"是全部都吃了，还是某几种？" |
| 用户说"停药/删除/不吃了" | 先复述药名确认，等用户回复后才执行 |
| 用户说"跳过" | 先问原因（不舒服？医生建议？），再执行 |
| 用户指令明确（只有一种药/说了药名） | 直接执行，不啰嗦 |

## 流式输出（SSE Streaming）

### 原理

DeepSeek API 支持 Server-Sent Events（SSE），AI 的回复不是一次性返回，而是一个 token 一个 token 地"流"回来。

```
前端发送请求（stream: true）
         │
         ▼
DeepSeek 开始生成
  → data: {"choices":[{"delta":{"content":"打"}}]}
  → data: {"choices":[{"delta":{"content":"卡"}}]}
  → data: {"choices":[{"delta":{"content":"成"}}]}
  → data: {"choices":[{"delta":{"content":"功"}}]}
  → data: [DONE]
```

### 前端处理

```typescript
const reader = response.body.getReader()
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  // 解析 SSE 数据，提取 delta.content
  // 逐字追加到界面上（打字机效果）
}
```

### 特殊处理：工具调用也是流式的

AI 返回 tool_calls 时，参数 JSON 也是分片到达的，需要拼接：

```
delta: { tool_calls: [{ index: 0, function: { arguments: '{"med' } }] }
delta: { tool_calls: [{ index: 0, function: { arguments: '_name"' } }] }
delta: { tool_calls: [{ index: 0, function: { arguments: ':"阿莫西林"}' } }] }
```

前端用 `toolMap` 按 index 累积拼接，最后 `JSON.parse` 得到完整参数。

## 记忆系统（extractMemories）

### 工作流程

每次对话结束后，后台异步分析最近 6 条消息，提取值得长期记住的信息：

```
用户："我对青霉素过敏，医生上周让我停了"
                │
                ▼  后台调 DeepSeek 分析
提取结果：[
  { type: "medical_note", key: "allergy", value: "青霉素过敏", confidence: 0.95 },
  { type: "medical_note", key: "recent_change", value: "医生上周停用青霉素", confidence: 0.90 }
]
                │
                ▼  置信度 > 0.85 → 写入 user_memory 表
                │
                ▼  下次对话时自动注入用户画像
```

### 设计要点

| 规则 | 原因 |
|------|------|
| 只提取置信度 > 0.85 的 | 避免把 AI 猜测的内容当事实记住 |
| 只提取用户明确说出的事实 | 不推断、不猜测 |
| 不记录临时操作 | "今天打卡了"这类不值得长期记 |
| 不重复已有信息 | 药品基本信息已在数据库里 |
| 异步执行，不阻塞 | 用户体验不受影响 |

### 记忆类型

| 类型 | 示例 |
|------|------|
| `medical_note` | "对青霉素过敏"、"医生建议饭后服用" |
| `preference` | "不喜欢早上吃药"、"希望回复简短些" |
| `habit` | "通常晚上 10 点吃药" |
| `context` | "下周要做体检" |

## 语音交互

### 语音输入（Speech-to-Text）

使用浏览器原生 Web Speech API：

```
用户按住麦克风 → 浏览器录音 → 实时识别中文 → 识别结果填入输入框 → 自动发送
```

- 支持中间结果实时显示（用户说话时就能看到文字）
- 识别完成后自动提交，无需手动点发送

### 语音输出（Text-to-Speech）

使用 Edge TTS（微软神经语音），通过 WebSocket 直连微软 TTS 服务：

```
AI 流式输出文字
    │
    ▼  SentenceDetector 检测句子边界（句号/问号/换行）
    │
    ▼  一句话完成 → 立即发送给 Edge TTS
    │
    ▼  WebSocket 合成 MP3 音频
    │
    ▼  TTSPlayer 音频队列按顺序播放
```

**关键设计：**
- **流式朗读**：不等全部文字输出，检测到一句话（句号/问号）就开始合成，边打字边说话
- **并行合成**：前一句在播放时，后续句子已经在合成，减少等待间隔
- **音频队列**：保证多段音频按正确顺序播放
- **兜底方案**：Edge TTS 不可用时自动退回浏览器内置语音

## 为什么这么设计（面试常问）

**Q: 为什么用 DeepSeek V3 而不是 R1？**
> V3 支持 function calling（工具调用），R1 是推理模型，输出的是思维链文本，不支持结构化工具调用。用药助手需要执行打卡、查库存等操作，必须用 V3。

**Q: 为什么要做对话历史压缩？**
> LLM 有上下文窗口限制（token 数），且 token 越多、响应越慢、成本越高。压缩旧对话为摘要能在保留上下文的同时控制 token 用量。阈值设为 4000 tokens，保留最近 6 轮完整对话。

**Q: 为什么工具在前端执行而不是后端？**
> 工具操作的数据（药品、打卡记录）都在 Supabase，前端已经有完整的数据层（Pinia store）。在前端执行可以立即更新 UI 状态，体验更流畅。敏感操作（如 API Key）才放到 Edge Function。

**Q: 为什么 TTS 要按句子切分而不是整段合成？**
> 整段合成要等 AI 输出完全部文字才能开始，用户等待时间长。按句子切分实现"边打字边说话"，第一句话出来就开始朗读，体验接近真人对话。

**Q: 记忆系统为什么要设置置信度阈值？**
> 防止 AI 把推测当事实记住。比如用户说"今天胃不舒服"，AI 可能推测"用户有胃病"，但这不一定对。只记录置信度 > 0.85 的明确陈述，保证记忆的准确性。
